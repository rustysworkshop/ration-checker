Startup self-tests

Before putting this tool in front of a client, or after loading a fresh copy on a new machine, run the self-test suite. It verifies:

  • LP solver is loaded and producing valid output on analytical test cases
  • Constraint compliance: any solution labelled "feasible" actually satisfies every constraint within numerical tolerance
  • Cost-floor correctness: hand-computed optima match solver output
  • Dominated-ingredient rejection: expensive substitutes for cheaper feeds don't appear in solutions
  • Regression baselines: three known scenarios produce expected results within 5% cost tolerance
Takes ~2 seconds. Results appear below.

If any test fails, the tool may still be usable but a specific behaviour has changed from the baseline. Read the failure details and decide whether the change is intended. For advanced debugging, open DevTools (F12) and run runGoldenTests() in the console.

Main workflow

The primary path — used 80% of the time — is analyse existing ration → build pellet to fill gaps:

  1. Section 1 — set animal class, stage, and milk yield (for lactating). Targets populate automatically from CSIRO feeding standards. Override any target manually if your animals' requirements differ from table values.
  2. Section 2 — activate the feed ingredients currently in the ration (click the toggle switches). Library is collapsed by default to save space.
  3. Section 3 — enter kg as-fed for each active ingredient. "as fed" means the weight your scales read, not dry-matter. The tool converts using each ingredient's DM%.
  4. Section 4 — click Analyse ration. Shows current nutrient profile vs targets, identifies gaps, flags acidosis/PUFA/urea risks, displays Ca:P ratio.
  5. Section 5 — click Build pellet to fill gaps. Auto-populates pellet spec based on gap analysis. Click Optimise pellet to run the LP solver. If infeasible, banner appears with specific "reduce X feedstuff" suggestions.

Alternative path (Section 6): build a full TMR from scratch without an existing-ration analysis step. Less commonly used. Requires manually activating a reasonable set of ingredients and setting targets.

Non-obvious behaviours

Two ingredient libraries

The ration library (Section 2) and the pellet library (Section 5 within the pellet spec panel) are separate. Ration library contains forages, byproducts, and common farm-gate feeds. Pellet library contains commercial pellet ingredients including mineral supplements (limestone, DCP, salt). If you can't find an ingredient, check both.

Inclusion-limit semantics differ between libraries

In the ration library, maxInc means absolute kg DM per head per day. In the pellet library, maxInc means % of pellet DM (so a 20% max on canola in a 6 kg pellet = 1.2 kg DM). This is intentional but can confuse new users editing a library.

ME semantics switch between modes

In pellet mode, ME targets are density (MJ/kg DM). In TMR/full-ration mode, ME targets are total MJ per head per day. Switching modes via the top toggle rewrites these fields — don't be surprised if ME min changes from 11 to 160 when you switch.

Ingredient costs are stored verbatim

The $/tonne values shown in Section 2's ingredient library are the actual values used by the LP objective. The tool does not apply any scaling factor — displayed costs are stored costs. If you need commercial sensitivity when sharing the file, edit the costs manually before distribution. Scaling all costs by a uniform constant would not change which ingredients the LP picks (LP objectives are scale-invariant under positive constants), only the reported dollar figure.

LP-only solver (v8)

The LP (linear programming) solver is the sole production path for both pellet and full-ration optimisation. When the LP is infeasible, the tool shows an infeasibility banner with specific relaxation guidance and produces no numerical output — producing plausible-but-spec-violating numbers was deemed worse than forcing the user to resolve the conflict. The gradient solver is retained as an emergency escape hatch only, accessible via browser console: window.USE_LP_PELLET = false or window.USE_LP_RATION = false. Gradient output may silently violate constraints; do not use for commercial decisions.

Pellet starch has a min ceiling but no max ceiling

The gap-loader caps p_starch_min at a realistic 55% — wheat is 65% starch, barley 52%, so a cereal-dominated pellet with some mineral headroom lands around 45-55%. When the base-ration starch gap would require more than this (e.g. 2.7% starch forage base with 24% target), the impossibility warning fires above the pellet spec card with a suggested larger pellet DM that would close the gap. The pellet starch MAX stays uncapped (99) — excess pellet starch on a pasture base is generally desirable (energy); acidosis risk is handled by the traffic-light banner in the analyse panel. Override p_starch_max manually if you want a hard ceiling for a specific formulation.

Nutrient-specific tolerance bands (v8)

The gap-loader uses a 5% relative tolerance for macros (CP, ME, NDF, ADF, fat, starch, RDP, UDP) and 2% for minerals (Ca, P, Mg, Na, S). Rationale: macro measurement variance from NIR is wider and clinical consequences of narrow over/undershoot are mild; mineral analysis is more precise and clinical consequences of deficiency are real (hypocalcaemia, hypomagnesaemia, salt starvation, S-limited MCP). When the base ration is within tolerance of target, the gap-loader treats it as "no gap" and sets the pellet min to the natural floor instead of chasing measurement-noise-level shortfalls.

Gap-loader impossibility warnings

When the base ration has a gap that mathematically requires the pellet to exceed a physically realistic nutrient density (CP > 30%, Ca > 3.5%, P > 1.8%, starch > 55%), a warning panel appears at the top of the pellet spec card explaining why the spec is unrealistic and suggesting a pellet DM increase that would close the gap at a practical concentration. The spec fields still populate (clamped at the ceiling) so the LP can produce a best-feasible output — the warning is advisory, not blocking.

UDP enforced at pellet level from Section 1's udp_min (v8)

Section 1's udp_min (total-ration target) is now propagated to the pellet LP. The pellet LP derives the pellet-level UDP% needed so base + pellet hits the total-ration target, using the same math as the CP gap-loader. No separate p_udp_min input exists — the pellet UDP requirement is auto-derived from the animal's total-ration UDP need and the analysed base ration. If base already exceeds UDP target, the pellet UDP constraint is inactive (0).

Total-ration column in pellet output (v8)

All three pellet output tables (Proximate, Minerals, AA) show Pellet and Total ration columns side-by-side. Total-ration values are DM-weighted averages of base + pellet for most nutrients, CP-weighted for amino acids, and computed from total kg for ratios (Ca:P). If no base ration has been analysed first, the Total ration column shows "—". Pellet-only values remain for pellet-spec discipline; total-ration values are for animal-nutrient gut-checking.

Constraint-check tolerance

The OK/FAIL badges on constraint tables use a 0.5%-of-value tolerance (capped at 0.1 absolute). This absorbs LP-solver precision drift — without it, a solution at 0.8594% Ca would show FAIL against a 0.86% min, when really the LP satisfied the constraint to 1e-8 internal precision. The tolerance is an order of magnitude tighter than typical NIR feed-analysis variance.

Ca:P ratio displayed but not directly constrained

The tool shows Ca:P ratio in the analyse and pellet output panels, and compares it to the target derived from your Section 1 Ca and P targets. The LP doesn't directly constrain the ratio — it hits individual Ca and P minimums. If both are at their minimums, the ratio falls where it falls. For lactating cows, target 1.4–1.8:1; the tool flags deviations outside ±15%.

Two separate nitrogen alerts

The tool shows both an Urea/NPN inclusion alert and a Rumen NH₃ load (est.) alert — they're checking different things. The NPN inclusion alert only tracks ingredients flagged as NPN (urea, ammonium salts) against a urea-equivalent threshold — a narrow mill-floor safety check. The rumen NH₃ load uses per-ingredient soluble-protein (CNCPS Fraction A) values to estimate total rapidly-released N across all sources — NPN, soluble true protein from fresh pasture, silage, legumes (lupins, peas, faba beans), and oilseed meals — and compares that against MCP capture capacity. A diet with zero urea can still show high rumen NH₃ load if it's heavy on fresh ryegrass flush or legume protein. Both metrics are useful; don't rely on just one.

Troubleshooting

Infeasibility banner appears (pellet mode)

Means the pellet LP cannot produce a pellet that simultaneously meets every constraint you've set. The banner's top recommendation is to reduce a specific base-ration ingredient (e.g. oaten hay) that's diluting a target nutrient. Try it via the Verify with LP button. If no single-ingredient reduction works, expand the "Or relax pellet constraint" section for constraint-level suggestions. In v8 the tool produces no output at all when infeasible — no spec-violating plausible numbers — you must resolve the conflict before the LP will produce a pellet. Genuinely infeasible scenarios usually mean the base ration is too heavy on low-quality forage relative to the animal's requirements.

Infeasibility banner appears (ration / TMR mode)

Same LP-only behaviour as pellet mode. When a full-ration LP is infeasible, output is cleared and the banner appears in the ration output panel. Most common cause in TMR mode is mineral-min enforcement (Mg/Na/S from Section 1 defaults) with a forage-only ingredient selection — the banner detects this and suggests either adding mineral supplements from the pellet library or zeroing the mineral mins for a forage-only study ration. Less common: AA mins (Lys/Met as %CP) binding against a forage that can't supply them.

Spurious FAIL badges on constraints

Every constraint-check table uses a 0.5% tolerance. If you see a FAIL where the displayed value equals the min, re-run the optimise (the LP can drift slightly between runs). A persistent FAIL means the LP genuinely could not meet that constraint — check the infeasibility banner above the results.

Odd ingredient choices in LP output

The LP picks least-cost feasible formulations. If it includes an ingredient you wouldn't expect, check: (a) is the stored $/tonne realistic and up-to-date for your local supplier? (b) are the nutrient values (CP/ME/starch) correct? (c) is there a forced minimum inclusion via minInc? The LP is mathematically optimal given its inputs; unexpected output usually means input data needs updating.

Pellet cost seems high

Check whether the LP is being forced into expensive minerals (DCP, synthetic AA, limestone) by tight Ca/P/Lys minimums — if so, the pellet is earning that cost. Temporarily relax individual mineral minimums to test sensitivity. Also verify the stored ingredient costs are realistic for your supply — replace the defaults with your actual contract prices before running formulations for real decisions.

Self-tests fail after updating an ingredient library

The regression tests in the self-test suite have baselines pinned to specific ingredient data. If you replace a default ingredient's nutrient values (e.g. updating Barley CP from 11 to 12), regression tests may fail even though the tool is working correctly. Review the failure details — if the deviation is proportional to your data update, the tests are correctly detecting the input change, not a bug in the tool.

Data provenance & limitations

Default nutrient values for ingredients come from INRA Feed Tables (2018), NRC Nutrient Requirements of Dairy Cattle (2021, 8th ed.) and published Australian feed-testing averages. Amino-acid profiles use the NRC 2001 MCP-AA model. Animal requirement tables are based on CSIRO Feeding Standards for Australian Livestock (ruminants).

For commercial formulation, always update ingredient nutrient values from your own NIR/wet-chem feed analyses. Generic defaults are starting points, not substitutes for measured data. A 15% error in CP or starch on the majority ingredient cascades through every downstream calculation.

Limitations:

  • The LP finds mathematically optimal formulations given the constraints and data provided. It cannot validate that the nutrient data is accurate.
  • Amino-acid as %CP is handled via iterative linearisation (two-pass); may under-iterate for CP targets that diverge substantially from what the LP actually produces.
  • The tool does not model rumen dynamics, passage rate, digestibility interactions, or particle-size effects. Output is a nutrient summation; biological response to a formulation is not predicted.
  • peNDF floors are heuristic (4 kg forage DM for lactating dairy, 3.5 for others). Actual rumen-function requirements vary with particle length and chop quality.
  • Stored ingredient costs are defaults; for real formulations, replace them with your actual contract or spot prices. The tool's cost output is only as accurate as its cost inputs.
  • Soluble-protein (Fraction A) values used for the rumen NH₃ load calculation come from CNCPS (Sniffen et al. 1992, updated Van Amburgh 2015), INRA Feed Tables 2018, and NRC 2001/2021. They are first-pass ingredient-library values — real soluble-protein content varies with maturity, ensiling quality, and processing. If you have NIR or wet-chem soluble protein analyses for your feeds, override the library values directly via ingredients[i].solubleN in the browser console.
  • Rumen NH₃ thresholds (0.10 and 0.25 kg/day protein-equivalent excess soluble N) are heuristic, calibrated to NRC 2001 RDP-MCP balance recommendations at dairy-cow scale. Sheep and growing cattle run smaller MCP pools; the absolute thresholds over-flag low-DMI classes. The ratio (Soluble-N:MCP) is scale-invariant and a better comparator across animal classes — values under 1.0 indicate microbial capture exceeds release.

Quick reference

Console commands (open DevTools with F12 → Console tab):

  • runGoldenTests() — run full self-test suite, print results to console
  • runGoldenTests({silent:true}) — run silently, return result object
  • window.USE_LP_PELLET = false — DEBUG ONLY: fall back to gradient solver for pellet (may violate constraints)
  • window.USE_LP_RATION = false — DEBUG ONLY: fall back to gradient solver for full ration (may violate constraints)
  • solvePelletLP(pelletIngredients.filter(i => i.active), {...}) — direct LP call

Print / export: browser's Print to PDF (Ctrl+P / Cmd+P). The @media print CSS is tuned to produce a clean formulation report without UI chrome.

Save library: Section 2 has Save library (downloads JSON of current ingredient set, pellet library included) and Load library (restores from JSON). In v8 all save filenames are date-stamped (e.g. ration-formulator-v8-2026-04-20.html, feed-library-v6-2026-04-20.json) so saving on different days preserves dated backups.

Hard reset: close and reopen the file — it's single-HTML with no browser-storage persistence by default, so each open is a clean state with the default libraries.
